## Phase 1：技术验证 Spike（当前阶段）

> **目标**：证明"刮擦"这个核心手感是可行的，且足够爽。
> **判断标准**：一个非游戏从业者拿到手机，刮 30 秒后说"好解压"。
> **时间预算**：1~2 周。超过则说明技术方案选型有问题，立刻换方案。

**此阶段禁止做的事**：写任何游戏逻辑、数值、UI、故事内容。

### 1-A. Scratch 渲染器技术选型（最高优先级）

研究并决定以下两个方案之一：

**方案 A：`RenderTexture` + 画刷擦除（推荐）**

- 原理：将一张"污垢遮罩图"渲染到 RenderTexture，用手指触点为圆心，将圆形区域的 Alpha 值清除为 0（透明），露出底图
- 优点：性能极高，手感丝滑，支持任意形状画刷，支持粒子特效
- 缺点：需要理解 Cocos 的 RenderTexture 和自定义 Material 管线
- 关键 API：`RenderTexture`，`Camera` 对准 RenderTexture，`Sprite` 使用自定义 Material 做 Alpha 混合

**方案 B：`Graphics` 组件 + `Mask`（备选）**
- 原理：用 Graphics 在 Mask 节点上画白色圆形，Mask 遮罩底图节点
- 优点：实现简单，无需自定义 Shader
- 缺点：当笔触密度高时（快速划动）性能急剧下降，DrawCall 爆炸

**Spike 验收标准**：
- [ ] 手指快速划动时，无明显卡顿（60fps 维持）
- [ ] 画刷边缘有柔和的羽化效果（不是硬边圆圈）
- [ ] 可以正确计算"已擦除面积百分比"（用于触发鉴定）
- [ ] 擦除操作在手机真机上流畅（不只是编辑器预览）

### 1-B. 基础音效反馈

- [ ] 接入 AudioManager，绑定一个"沙沙"刮擦音效
- [ ] 触点移动速度影响音效音量/频率（快划 = 大声）
- [ ] 不需要完整音效库，一个临时音效文件即可

## Phase 2：MVP 单局闭环（近期）

> **目标**：玩家能完整体验"获取文物 → 刮擦清洁 → 鉴定估值 → 卖出/捐献 → 金币增加"这一完整循环。
> **判断标准**：一个完整的单局，从头到尾没有死路，数值有意义。
> **时间预算**：3~4 周。

### 2-A. 核心数据模型

- [ ] `GameData.ts`：定义运行时数据结构
  - `coins`（金币）
  - `reputation`（声望，捐献获得）
  - `artifactSlots`（当前持有的待处理文物列表）
  - `museumCollection`（已入馆的文物 ID 列表）
- [ ] `ArtifactConfig`（配表）：文物的静态数据
  - `id`，`name`，`rarity`（普通/稀有/传奇/国宝）
  - `baseValue`（基础估值金币）
  - `historyStory`（历史故事文本，可为空）
  - `isNationalTreasure`（是否为流失国宝，禁止出售）
- [ ] `ArtifactInstance`（运行时）：一件文物的当前状态
  - `configId`，`cleanPercent`（清洁进度 0~1），`condition`（品相：完好/轻微损坏/破损）

### 2-B. 单局流程串联

- [ ] `MainWindow` 承载核心玩法区（刮擦组件 + 顶部资源栏）
- [ ] 刮擦进度达到 90% 时，自动触发"鉴定"流程
- [ ] 鉴定结果弹窗（`AppraisalPopup`）：
  - 显示文物名称、估值、品相
  - 两个按钮：【卖出（金币）】 / 【捐献（声望）】
  - 国宝级文物：【卖出】按钮禁用，强制捐献（文案自然，不说教）
- [ ] 卖出/捐献后，更新 `GameData`，播放金币飞入动画
- [ ] 流程结束，生成下一件文物（暂时随机从配表中取）

### 2-C. 工具升级系统（最小版本）

- [ ] 3 种工具档次：刷子 / 抛光机 / 高压水枪
- [ ] 每种工具影响：画刷半径大小 + 刮擦音效
- [ ] `UpgradePopup`：简单的升级面板，显示当前工具和升级费用
- [ ] 不需要复杂的树状升级，线性升级即可

### 2-D. 博物馆基础展示（最小版本）

- [ ] `MuseumWindow`：一个滚动列表，展示已捐献的文物
- [ ] 每件文物显示：图标、名称、入馆日期
- [ ] 暂不需要 3D 展馆、自定义布局等
- [ ] 图鉴"空格"显示为黑色剪影（集满系列奖励的预埋）

### 2-E. 游戏故事导入（最小版本）

- [ ] 开场对话：外公手记第一页（3~5 句话，奠定基调，不宏大）
- [ ] 房东催租事件（触发第一次刮擦的动机）
- [ ] 不需要完整剧情系统，硬编码触发即可

## Phase 3：内容填充（中期）

> **目标**：让游戏有足够的内容支撑玩家玩 2~4 周不腻。
> **前提**：Phase 2 的单局闭环已被真实玩家验证"好玩"。

### 3-A. 文物内容扩充

- [ ] 至少 30 件文物（跨越 5 个历史时期）
- [ ] 3~5 种文物材质（陶瓷/青铜/玉器/书画/金器），对应不同刮擦手感和音效
- [ ] 3 件"流失国宝"（有真实历史原型，捐献触发专属动画）
- [ ] 套装系统：集齐同一系列解锁全局 Buff

### 3-B. 出海打捞地图（探索层）

- [ ] 手绘风格航海地图（外公手记形式）
- [ ] 4~6 个打捞区域，不同区域出产不同历史时期的文物
- [ ] 出海消耗"燃料"（体力系统，限制频率）
- [ ] 随机叙事事件卡片（文字+插画，3 选 1 分支）

### 3-C. NPC 送宝系统（喜剧层）

- [ ] 每日刷新 1~2 个 NPC 登门
- [ ] 包含"微波炉专用"梗级别的搞笑文物（教程兼容）
- [ ] 简单砍价小游戏（影响收购价格）

### 3-D. 鉴定问答系统（深度层）

- [ ] 鉴定流程加入 3 道选择题（依据刮擦时观察的细节）
- [ ] 答题结果影响文物最终估值（完美鉴定 +30%）
- [ ] 题库绑定文物 Config，不同文物有不同题目

### 3-E. Idle 自动化（放置层）

- [ ] 雇佣"学徒"（消耗金币/声望）
- [ ] 学徒自动处理普通品质文物（玩家只需处理稀有以上）
- [ ] 离线收益（关闭游戏后，学徒持续产出）

### 3-F. 配表工具

- [ ] Excel/CSV 格式的文物配表（策划可独立维护）
- [ ] 构建时自动转换为游戏可读的 JSON
- [ ] 考虑使用 Google Sheets 作为策划协作工具

### 3-G. 资源管线

- [ ] 文物图片命名规范（`artifact_{id}_{state}.png`）
- [ ] 音效命名规范和分类（`sfx_scratch_soft`，`sfx_reveal_rare` 等）
- [ ] 资源打包策略（哪些进 Bundle，哪些走 Resources 热加载）

## Phase 4：社交 & 联机（远期）

> **前提**：有稳定的单机玩家基础，且有后端工程师资源。
> **绝对不要在 Phase 2 完成前碰这一块。**

- [ ] 账号系统（微信授权登录）
- [ ] 玩家数据云端同步
- [ ] 博物馆互访（异步，查看好友博物馆）
- [ ] 文物碎片交换（类 Monopoly Go 贴纸交换）
- [ ] 声望排行榜（捐献总价值排名）
- [ ] 全服协作事件（限时，制造 FOMO）
- [ ] 后端接入（Golang 服务，高并发保障）

## 永远在做的事（横切关注点）

无论处于哪个阶段，以下事项持续进行：

- **真机测试**：每个功能完成后，立刻在真机上跑一遍，不要只看编辑器预览
- **性能监控**：DrawCall 和 Overdraw 不能因为新功能失控，每周检查一次
- **存档兼容性**：`SaveManager` 的数据结构变更，必须做版本迁移，不能让老存档崩溃
- **剪枝**：每进入新阶段前，重新评估当前阶段所有未完成项的必要性，不重要的直接删除
